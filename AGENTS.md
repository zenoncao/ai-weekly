# Repository Guidelines

## Project Shape

This is a small Astro static site for a Chinese AI weekly blog.

- Source pages live in `src/pages/`.
- Weekly issue data lives in `src/data/issues.ts`.
- Global styling lives in `src/styles/global.css`.
- Built output lives in `dist/` and should be treated as generated unless the task explicitly asks for static export artifacts.

## Commands

- `npm run dev` starts the Astro dev server.
- `npm run build` runs the production build and type checks Astro pages.
- `npm run preview` previews the built site.

Run `npm run build` after code, data-shape, route, or CSS changes unless the task is purely textual and does not affect buildable files.

## Content Rules

The site is source-heavy editorial content. Do not invent facts, numbers, publication dates, author names, quotes, product names, or URLs.

When adding or editing articles in `src/data/issues.ts`:

- Verify each source URL and publication date if the article is current, recent, or otherwise unstable.
- Keep `summary`, `why`, and `factOpinionInference` distinct: facts belong in the summary, editorial judgment belongs in `why`, and source/inference separation belongs in `factOpinionInference`.
- Use the existing Chinese editorial voice: direct, analytical, and specific.
- Keep confidence labels limited to the existing union: `高`, `中`, `低`.
- Keep signal labels limited to the existing union: `高`, `中高`, `中`.
- Keep categories limited to the `Article["category"]` union unless the page rendering and styles are updated at the same time.

## Data And Routing

`issues` is ordered newest first, and `latestIssue` currently resolves to `issues[0]`.

Each issue needs:

- a stable `slug`, normally an ISO date such as `2026-06-09`;
- a `date` string;
- a `range` string;
- a `deck` and `editorial`;
- an `articles` array.

Issue detail pages are generated by `src/pages/issues/[slug].astro` via `getStaticPaths()`. If a new issue is added to `issues`, the route is generated automatically.

## Style Guidelines

The visual design is a dense dark editorial interface with restrained cards, mono metadata, and colored signal accents. Preserve the current design language unless the user explicitly asks for a redesign.

- Keep card radii at or below the existing `--radius: 8px`.
- Avoid decorative SVGs, gradient blobs, or marketing-style hero sections.
- Check mobile layout after large CSS changes, especially the sticky header and category nav offsets.
- Do not introduce a one-hue palette; the current accents intentionally mix cyan, lime, gold, coral, violet, and blue.

## Editing Boundaries

- Prefer small, scoped changes.
- Do not edit `package-lock.json` unless dependencies actually change.
- Do not commit generated `dist/` changes unless the user asks for build output to be updated.
- Use existing Astro and TypeScript patterns before adding new abstractions.
